# Kairos Parts — Day 9 Manager CRM / Admin Panel

## Що реалізовано

Day 9 замінив CRM skeleton на робочу admin area для ролей `MANAGER` і `ADMIN`.

Реалізовано:

- role-aware `/admin` layout;
- CRM dashboard;
- список заявок із фільтрами та пошуком;
- картку заявки;
- зміну статусу заявки;
- запис `RequestStatusHistory`;
- призначення відповідального менеджера для `ADMIN`;
- внутрішні CRM-коментарі;
- клієнтську базу;
- перегляд техніки й документів клієнта;
- CRM API для заявок, статусів, призначення, коментарів і клієнтів.

## CRM dashboard

Сторінка `/admin` показує:

- всього заявок;
- нові заявки;
- заявки в роботі;
- заявки, що очікують погодження;
- завершені заявки;
- скасовані заявки;
- заявки за останні 7 днів;
- заявки за останні 30 днів;
- останні заявки;
- розподіл за статусами.

Дані беруться з реальної БД. Якщо `DATABASE_URL` не налаштований, показується blocker, а не fake-data.

На Day 9 `MANAGER` бачить загальний CRM-потік заявок. Обмеження тільки призначеними йому заявками не ввімкнено і має бути окремим рішенням.

## Список заявок

Сторінка `/admin/requests` показує:

- номер заявки;
- дату створення;
- клієнта або гостя;
- телефон;
- джерело;
- категорію;
- тип техніки;
- статус;
- відповідального менеджера;
- дату останнього оновлення;
- посилання на картку заявки.

## Фільтри й пошук

Доступні фільтри:

- статус;
- джерело;
- менеджер;
- категорія;
- дата від;
- дата до.

Пошук працює по:

- номеру заявки;
- телефону;
- імені гостя;
- назві компанії;
- VIN / серійному номеру;
- контактному імені клієнта;
- компанії клієнта;
- телефону клієнта;
- email клієнта.

## Картка заявки

Сторінка `/admin/requests/[id]` показує:

- номер заявки;
- джерело;
- дату створення;
- дату оновлення;
- поточний статус;
- відповідального менеджера;
- контактні дані;
- опис потреби;
- категорію;
- підкатегорію;
- виробника;
- тип техніки;
- модель;
- VIN / серійний номер;
- прив'язану техніку;
- файли заявки;
- документи заявки;
- public status URL;
- внутрішні коментарі;
- історію статусів.

Внутрішні коментарі показуються тільки в CRM/admin view.

## Зміна статусу

`MANAGER` і `ADMIN` можуть змінити статус заявки.

Підтримані статуси:

- `NEW`;
- `IN_PROGRESS`;
- `OFFER_PREPARING`;
- `WAITING_APPROVAL`;
- `ORDERED`;
- `IN_DELIVERY`;
- `COMPLETED`;
- `CANCELLED`.

Після зміни статусу:

- оновлюється `Request.status`;
- створюється запис `RequestStatusHistory`;
- фіксується `changedByUserId`.

Telegram/email notifications на Day 9 не реалізовувались.

## Призначення менеджера

`ADMIN` може призначити відповідального менеджера на заявку.

Список кандидатів береться з `User` з ролями:

- `MANAGER`;
- `ADMIN`.

`MANAGER` бачить поточного відповідального, але не призначає менеджера на Day 9. Self-assign не реалізований і має бути окремим правилом workflow.

## Внутрішні коментарі

`MANAGER` і `ADMIN` можуть додавати внутрішні коментарі.

Зберігається:

- `requestId`;
- `authorId`;
- `message`;
- `internal = true`;
- `createdAt`.

Клієнтський кабінет ці коментарі не читає і не показує.

## Клієнтська база

Сторінка `/admin/clients` показує:

- ім'я / контактну особу;
- компанію;
- телефон;
- email;
- тип клієнта;
- кількість заявок;
- кількість техніки;
- дату останньої заявки;
- посилання на детальну сторінку клієнта.

Редагування клієнтів не входить у Day 9.

## Техніка й документи клієнта

Менеджер бачить техніку й документи:

- у картці заявки, якщо заявка прив'язана до техніки;
- у картці заявки через пов'язаний `ClientProfile`;
- на `/admin/clients/[id]`.

Private storage paths не показуються. Якщо `fileUrl` не налаштований, CRM показує metadata.

## Розмежування прав

`MANAGER` може:

- відкривати CRM dashboard;
- бачити заявки;
- відкривати картку заявки;
- змінювати статус;
- додавати внутрішні коментарі;
- бачити клієнтів;
- бачити техніку й документи.

`ADMIN` може все, що `MANAGER`, а також:

- призначати відповідального менеджера;
- відкривати `/admin/settings`;
- відкривати `/admin/categories`;
- відкривати `/admin/manufacturers`;
- викликати admin-only catalog API placeholders.

## Перевірка доступу

Доступ перевіряється на кількох рівнях:

- middleware для `/admin/*`;
- `requireCrmSession` у admin layout/pages;
- `requireAdminSession` для admin-only pages;
- `getCrmApiSession` у CRM API;
- `getAdminApiSession` для admin-only API;
- UI приховує admin-only navigation для `MANAGER`;
- assign form доступна тільки `ADMIN`.

`CLIENT` і неавторизовані користувачі не мають доступу до CRM endpoints.

## API

Реалізовано:

- `GET /api/admin/requests`;
- `GET /api/admin/requests/[id]`;
- `PATCH /api/admin/requests/[id]/status`;
- `PATCH /api/admin/requests/[id]/assign`;
- `POST /api/admin/requests/[id]/comments`;
- `GET /api/admin/clients`.

Catalog POST placeholders лишились без повного CRUD, але тепер мають `ADMIN` API guard.

## Що працює після Day 9

- Менеджер може працювати із заявками в CRM.
- Менеджер бачить клієнтів, техніку, файли й документи.
- Менеджер може змінювати статуси та додавати внутрішні коментарі.
- Адміністратор може призначати відповідального менеджера.
- Адміністратор має доступ до admin-only розділів.
- API блокує неавторизовані ролі.
